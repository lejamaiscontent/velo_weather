# Знания о проекте: накопленные факты

## Open-Meteo: расписание обновления моделей

### Официальное расписание (из документации и substack)

| Модель | Прогоны (UTC) | Официальная задержка |
|--------|--------------|----------------------|
| ICON Global (DWD) | 00, 06, 12, 18 | ~2–3ч после прогона |
| ICON-EU (DWD) | 00, 03, 06... каждые 3ч | ~1–2ч после прогона |
| ICON-D2 (DWD) | каждые 3ч | ~1–2ч после прогона |
| ICON seamless | комбо: берёт наиболее детальную доступную | ~1–3ч после прогона |
| ECMWF IFS 0.25° (open data) | 00, 06, 12, 18 | +2ч к real-time (~6–8ч суммарно) |
| ECMWF IFS 9km (real-time) | 00, 06, 12, 18 | минимальная (~2–3мин на S3) |

**Итог для нашего использования** (icon_seamless + ecmwf_ifs025):
- ICON: новые данные реально могут появиться через ~3ч после прогона
- ECMWF: новые данные через ~6–8ч после прогона (используем open data tier)

### Что Open-Meteo НЕ возвращает в API
- Время инициализации модели (`model_init_time`) — в ответе отсутствует
- `generationtime_ms` — только время генерации HTTP-ответа, не прогона
- `times[0]` в hourly — всегда 00:00 UTC текущего дня, не время прогона

Единственный способ узнать, пришли ли новые данные — **сравнить снепшоты**.

## Алгоритм умного расписания фетча (реализован в app.py)

### Логика `_should_fetch_model(model, now)`

**Если известно время последнего обнаруженного обновления (`last_update_at`):**
1. Тихая зона: не трогаем модель пока `age < quiet_h`
2. После тихой зоны: поллинг каждые `poll_interval_min` минут

**Если история неизвестна (рестарт, первый запуск):**
1. Медленная фаза: поллинг каждые `initial_interval_min` минут
2. После `search_start_h` с первого фетча — переключаемся на быстрый поллинг каждые 15 минут

### Параметры по моделям

| Параметр | ICON seamless | ECMWF IFS 0.25° |
|----------|--------------|-----------------|
| `quiet_h` | 2ч45м | 10ч |
| `search_start_h` | 2ч31м (151мин) | 11ч |
| `initial_interval_min` | 50 мин | 170 мин (2ч50м) |
| `poll_interval_min` | 15 мин | 15 мин |

### Детектор изменений `_weather_changed(old, new)`

Сравниваем ~8 равноудалённых точек маршрута, каждые 6 часов прогноза:
- `max(diffs) > 0.5 м/с` по ветру
- `mean(diffs) > 0.15` (с весом ×2 для осадков)

Порог выбран с запасом: флуктуации аппроксимации порядка 0.1–0.2, реальное обновление прогона даёт отличия >>1 м/с.

### Лог-файл `weather_log.jsonl`

Каждая запись:
```json
{
  "t": "2026-06-16T21:38:37+00:00",
  "model": "icon_seamless",
  "new_run": true,
  "mode": "quiet_2.75h",
  "check": "probe",     // "probe" (дешёвая проверка) или "full" (полный фетч)
  "points": 8,          // сколько точек реально запрошено
  "last_update": null
}
```
(Поля `check`/`points` добавлены 17.06.2026; в старых записях их нет — читать через `.get()`.)

**Режимы (`mode`):**
- `slow_50min` / `slow_170min` — медленная фаза (нет истории)
- `search_15min` — быстрый поллинг в ожидании нового прогона
- `quiet_2.75h` / `quiet_10.0h` — тихая зона после известного обновления

### Дешёвая проба обновления (`_probe_model`, добавлено 17.06.2026)

Чтобы проверка «обновилась ли модель» не стоила полного фетча всей сетки (N≈52–114 точек),
фоновый цикл сначала делает **пробу**: запрашивает только `A = min(20, ceil(√N))` случайных
точек (≈8–11) и сверяет их с теми же индексами последнего полного снепшота тем же порогом
(`_diffs_exceed`). Экономия ~6–10× запросов на каждый поллинг.

- Проба строит сетку при `last_weather_km` (current_km на момент снепшота) — индексы пробы и
  снепшота гарантированно совпадают (`weather_query_segments` детерминирована).
- Результат: `False` — не обновилась, полный фетч пропускаем; `True` — обнаружено отличие →
  полный фетч; `None` — нет снепшота / сетка разъехалась / ошибка сети → полный фетч.
- Полный фетч (`_fetch_models`) делается только при `True`/`None`; он и обновляет кеш погоды
  для симуляции и выставляет `last_update_at`. После «чистой» пробы кеш не трогаем.
- В quiet-зоне поллинга нет вообще (`_should_fetch_model` → False), проба применяется в
  search/slow-фазах, где раньше каждый 15-мин поллинг стоил полного фетча.

## Анализ weather_log: первые результаты (17.06.2026)

Разбор накопленного `weather_log.jsonl` (14 записей, период 16.06 21:38 → 17.06 08:32 UTC).
**Выборка крошечная**, выводы предварительные.

### Чистка ложных срабатываний
- Старт сервера 16.06 21:38 (ICON+ECMWF) и **рестарт** 17.06 08:31 (ICON+ECMWF) — `new_run=true` ложно: нет предыдущего снепшота (`_model_sched` не переживает рестарт).
- Рестарт 08:31 опознан так: обе модели фетчатся одновременно, а ECMWF за 53 мин до этого был свежий (quiet 10ч) — планового фетча быть не могло ⇒ снепшот стёрся. Запись 08:32 (обе `false`) это подтверждает.
- Непрерывная сессия фактически одна: 16.06 21:38 → ~17.06 08:31 (~11ч).

### Реальные обнаружения

| Модель | Detected UTC | Окно появления | Точность |
|--------|-------------|----------------|----------|
| ICON  | 17.06 00:23 | 21:38→00:23 | ±2.75ч (выход из quiet) |
| ICON  | 17.06 04:08 | 03:53→04:08 | **±15 мин (search)** |
| ICON  | 17.06 06:53 | 04:08→06:53 | ±2.75ч (выход из quiet) |
| ECMWF | 17.06 07:38 | 21:38→07:38 | ±10ч (выход из quiet) |

### Выводы
1. **Детекции садятся ровно на край quiet-зоны** (00:23 = 21:38+2.75ч; 06:53 = 04:08+2.75ч; ECMWF 07:38 = 21:38+10ч — секунда-в-секунду). Значит новые данные появились *внутри* тихой зоны и там не отслеживались → латентность модели эти точки **не измеряют**.
2. **Единственное настоящее измерение — ICON @ 04:08** (поймано в search, окно 15 мин): появление ≈ 04:00–04:08 UTC ⇒ прогон 03Z ICON-EU/D2 + ~1ч. Подтверждает: `icon_seamless` обновляется по мелкой сетке (D2/EU ~каждые 3ч), не только 00/06/12/18.
3. **quiet=2.75ч для ICON, вероятно, длинновата** — D2/EU цикл ~3ч, на выходе из quiet почти всегда уже есть новое ⇒ едем на слегка устаревшем прогнозе.
4. **ECMWF при quiet=10ч не измерить** — одна точка с окном 10ч.
5. **Детектор `_weather_changed` даёт ложные негативы**: в 03:08 `new_run=false`, хотя 00Z к ~02:30 должен был выйти. Грубая выборка (каждые 6ч, порог 0.5 м/с) пропускает обновления, слабо двигающие ветер в опорных точках.

**Методический итог:** текущая схема почти не измеряет истинную задержку публикации — нужно сузить quiet либо принудительно входить в search вокруг расчётного `run+delay`, и логировать каждый поллинг с ожидаемым временем прогона.

## Поведение симуляции: дождь vs без дождя

- Обе симуляции считаются при каждом `simulate_only()` для каждой модели
- Разница ТОЛЬКО в `rain_threshold`: 0.5 мм/ч (с дождём) vs 9999 (без дождя)
- Planned stops (`planned_stop_budget_h`) одинаковы в обоих режимах — это ожидаемое поведение
- Реальный разброс финиша для ECMWF: ~2.5ч (пример 16.06: 04:09 UTC vs 01:42 UTC)
- Для ICON в тот же момент разница была 0 — дождевых стоянок нет по прогнозу

### Стоп-баннеры в UI (шаблон)
- 🌧 `rain-wait` (оранжевый) — `stop_reason` содержит «дождь»
- ☕ `planned` (серый) — любой другой `stop_reason` (плановый перерыв)
- 📍 `actual` (зелёный) — фактическая стоянка из `state.actual_stops`

## Архитектурные решения (причины)

### Почему `fetch_weather` и `simulate_only` разделены
Каждый UI-action (смена км, установка мощности, добавление стоянки) вызывает `simulate_only()` без запроса к API — экономим лимит Open-Meteo (бесплатный план, ~10к запросов/сутки).

### Почему `_fetch_lock`
Три одновременных вызова `fetch_weather` при старте (startup + set_power + force_recalc) давали 120+ параллельных HTTP-запросов к Open-Meteo → API отвечал 429. Блокировка вторых и третьих вызовов решила проблему.

### Почему нелинейная сетка точек
Первые 100 км от текущей позиции — точки каждые 4 км (детальный прогноз нужен скоро).
Дальше — плавно до 25 км (экономим запросы, прогноз всё равно теряет точность на горизонте 2–3 дней).

### Почему `_model_sched` не сериализуется в state.json
Состояние расписания (когда видели новый прогон) имеет смысл только в рамках текущей сессии. После рестарта начинаем заново — первый фетч всегда «НОВЫЙ ПРОГОН» (нет предыдущего снепшота для сравнения), потом входим в тихую зону. Это правильно: при рестарте мы итак получаем свежие данные.

## Архив статистики заезда (#26, добавлено 19.06.2026)

Цель — офлайн-отладка: совпадение прогнозов после обновления погоды, точность прогноза
скорости, точность перерасчёта мощности. Пишется на VPS, стягивается `scp` как логи.

**Папка:** `archive/<ride_id>/`, где `ride_id` = `start_time[:16]` с заменой `:`→`-`
(напр. `2026-06-20T05-00`). Управляется конфигом: `archive_enabled` (default true), `archive_dir`.

| Файл | Когда пишется | Содержимое |
|------|---------------|-----------|
| `meta.json` | один раз на заезд | config, GPX (имя+sha1+total_km), git-commit, модели — для реплея |
| `forecast.jsonl` | каждый `simulate_only()` | `t, trigger, current_km, eff_power`, по модели: `run_at, finish, finish_no_rain, grid[]` (компактный 10км-грид) |
| `position.jsonl` | каждый `update_position()` | факт `km/время`, `actual_speed_kmh, riding_h, calibrated_power, forecast_age_h, live_pred` (прогноз на этом км до пересчёта) |
| `power_log.jsonl` | калибровка (расширен) | `from/to_km, riding_h, elapsed_h, stop_h, prev_power, default_power, delta_w` |

**Триггеры `forecast`:** `startup` / `weather` (новый прогон) / `position` / `power` / `manual`.
diff соседних снимков с `trigger=weather` → насколько новый прогон сдвинул план по каждому км.

**Анализ (офлайн):** скорость — `position.actual_speed_kmh` vs `forecast.grid[].speed_kmh` на том
же км при разных lead'ах (join по km); мощность — стабильность `calibrated_power` по отрезкам +
ошибка ETA; погода-«истина» — прогноз с минимальным lead из `forecast.jsonl`.

**Защита:** все записи в try/except — архив не может уронить `simulate_only`/`update_position`.

### Стоп-гап: текущая погода (#30, 19.06.2026)

`weather_now.jsonl` — раз в 30 мин (`_bg_loop` → `_snapshot_current_weather`) снимок ТЕКУЩЕЙ
погоды по точкам маршрута: `{t, model, points:[{lat,lon,wind_ms,wind_dir,precip_mm_h,temp_c}]}`.
Значения берутся из актуального кеша прогноза через `core._interp(wp, now)` — **нулевой расход API**
(это свежайший доступный nowcast). Зачем: `forecast.jsonl` покрывает только окно заезда, а здесь
копится непрерывный ряд погоды для местности → погода-«истина» для сверки прогнозов и отладки
физики. Решение временное; «честный» вариант — отдельный фетч `current=` с Open-Meteo.
