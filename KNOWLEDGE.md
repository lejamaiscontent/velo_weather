# Знания о проекте: накопленные факты

## Open-Meteo: расписание обновления моделей

### Официальное расписание (из документации и substack)

| Модель | Прогоны (UTC) | Официальная задержка |
|--------|--------------|----------------------|
| ICON Global (DWD) | 00, 06, 12, 18 | ~2–3ч после прогона |
| ICON-EU (DWD) | 00, 03, 06... каждые 3ч | ~1–2ч после прогона |
| ICON-D2 (DWD) | каждые 3ч | ~1–2ч после прогона |
| ICON seamless | комбо: берёт наиболее детальную доступную | ~1–3ч после прогона |
| ECMWF IFS 0.25° (open data) | 00, 06, 12, 18 | +2ч к real-time (~6–8ч суммарно) |
| ECMWF IFS 9km (real-time) | 00, 06, 12, 18 | минимальная (~2–3мин на S3) |

**Итог для нашего использования** (icon_seamless + ecmwf_ifs025):
- ICON: новые данные реально могут появиться через ~3ч после прогона
- ECMWF: новые данные через ~6–8ч после прогона (используем open data tier)

### Что Open-Meteo НЕ возвращает в API
- Время инициализации модели (`model_init_time`) — в ответе отсутствует
- `generationtime_ms` — только время генерации HTTP-ответа, не прогона
- `times[0]` в hourly — всегда 00:00 UTC текущего дня, не время прогона

Единственный способ узнать, пришли ли новые данные — **сравнить снепшоты**.

## Алгоритм умного расписания фетча (реализован в app.py)

### Логика `_should_fetch_model(model, now)`

**Если известно время последнего обнаруженного обновления (`last_update_at`):**
1. Тихая зона: не трогаем модель пока `age < quiet_h`
2. После тихой зоны: поллинг каждые `poll_interval_min` минут

**Если история неизвестна (рестарт, первый запуск):**
1. Медленная фаза: поллинг каждые `initial_interval_min` минут
2. После `search_start_h` с первого фетча — переключаемся на быстрый поллинг каждые 15 минут

### Параметры по моделям

| Параметр | ICON seamless | ECMWF IFS 0.25° |
|----------|--------------|-----------------|
| `quiet_h` | 2ч45м | 10ч |
| `search_start_h` | 2ч31м (151мин) | 11ч |
| `initial_interval_min` | 50 мин | 170 мин (2ч50м) |
| `poll_interval_min` | 15 мин | 15 мин |

### Детектор изменений `_weather_changed(old, new)`

Сравниваем ~8 равноудалённых точек маршрута, каждые 6 часов прогноза:
- `max(diffs) > 0.5 м/с` по ветру
- `mean(diffs) > 0.15` (с весом ×2 для осадков)

Порог выбран с запасом: флуктуации аппроксимации порядка 0.1–0.2, реальное обновление прогона даёт отличия >>1 м/с.

### Лог-файл `weather_log.jsonl`

Каждая запись:
```json
{
  "t": "2026-06-16T21:38:37+00:00",
  "model": "icon_seamless",
  "new_run": true,
  "mode": "quiet_2.75h",
  "last_update": null
}
```

**Режимы (`mode`):**
- `slow_50min` / `slow_170min` — медленная фаза (нет истории)
- `search_15min` — быстрый поллинг в ожидании нового прогона
- `quiet_2.75h` / `quiet_10.0h` — тихая зона после известного обновления

## Поведение симуляции: дождь vs без дождя

- Обе симуляции считаются при каждом `simulate_only()` для каждой модели
- Разница ТОЛЬКО в `rain_threshold`: 0.5 мм/ч (с дождём) vs 9999 (без дождя)
- Planned stops (`planned_stop_budget_h`) одинаковы в обоих режимах — это ожидаемое поведение
- Реальный разброс финиша для ECMWF: ~2.5ч (пример 16.06: 04:09 UTC vs 01:42 UTC)
- Для ICON в тот же момент разница была 0 — дождевых стоянок нет по прогнозу

### Стоп-баннеры в UI (шаблон)
- 🌧 `rain-wait` (оранжевый) — `stop_reason` содержит «дождь»
- ☕ `planned` (серый) — любой другой `stop_reason` (плановый перерыв)
- 📍 `actual` (зелёный) — фактическая стоянка из `state.actual_stops`

## Архитектурные решения (причины)

### Почему `fetch_weather` и `simulate_only` разделены
Каждый UI-action (смена км, установка мощности, добавление стоянки) вызывает `simulate_only()` без запроса к API — экономим лимит Open-Meteo (бесплатный план, ~10к запросов/сутки).

### Почему `_fetch_lock`
Три одновременных вызова `fetch_weather` при старте (startup + set_power + force_recalc) давали 120+ параллельных HTTP-запросов к Open-Meteo → API отвечал 429. Блокировка вторых и третьих вызовов решила проблему.

### Почему нелинейная сетка точек
Первые 100 км от текущей позиции — точки каждые 4 км (детальный прогноз нужен скоро).
Дальше — плавно до 25 км (экономим запросы, прогноз всё равно теряет точность на горизонте 2–3 дней).

### Почему `_model_sched` не сериализуется в state.json
Состояние расписания (когда видели новый прогон) имеет смысл только в рамках текущей сессии. После рестарта начинаем заново — первый фетч всегда «НОВЫЙ ПРОГОН» (нет предыдущего снепшота для сравнения), потом входим в тихую зону. Это правильно: при рестарте мы итак получаем свежие данные.

## Известные ограничения и TODO

- Первый фетч после рестарта всегда помечается `new_run=true` (нет истории — ложно-положительный)

- `recalc_interval_min` в `config.json` больше не используется для фонового цикла (цикл умный), но оставлен для совместимости
