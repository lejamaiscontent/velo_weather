# Сверка физики по реальному прохождению (`analysis/`)

Отдельная офлайн-часть проекта: проверить и откалибровать физику (`core.solve_speed`) по
**реальному** заезду гонщика. Связана с задачами #2 (корректность скорости), #4 (усталость),
#31 (проверка трека по мощности).

**Логика:** на каждом отрезке реального трека известны
- фактическая скорость — из `<time>` + координат соседних точек;
- измеренная мощность — с power meter;
- уклон — из высот;

а ветер берём из снепшота погоды. Предсказываем скорость по физике и сравниваем с фактом;
опционально подбираем `CdA`/`Crr`, которые лучше всего объясняют реальные данные.

## Откуда брать данные

| Источник | Мощность? | Как получить |
|----------|-----------|--------------|
| **`.FIT`** (рекомендую) | да | Strava → активность → «···» → **Export Original** |
| `.TCX` | да (`<Watts>`) | экспорт из приложения/часов |
| `.GPX` | **обычно НЕТ** | Strava GPX-экспорт выкидывает ватты |
| Strava API streams | да | OAuth (токен — секрет, в `.env`, не в репо) |

Файлы заезда кладём в `analysis/data/` (каталог в `.gitignore` — приватные данные не коммитим).
Для `.fit` нужен `pip install fitparse`.

## Использование

```bash
# сводка по файлу (точки, время, есть ли мощность)
python analysis/parse_ride.py analysis/data/ride.fit

# сверка с физикой (ветер из ERA5-реанализа по датам трека)
python analysis/validate_physics.py analysis/data/ride.fit --mass 90

# подбор CdA/Crr под реальные данные
python analysis/validate_physics.py analysis/data/ride.fit --mass 90 --fit

# свой ветер (м/с, направление «откуда») или наш снепшот
python analysis/validate_physics.py analysis/data/ride.fit --weather const:3,270
python analysis/validate_physics.py analysis/data/ride.fit --weather snapshot:weather_now.jsonl
```

## Что считает

- `parse_ride.py` — `.fit/.tcx/.gpx` → нормализованные точки `(t, lat, lon, ele, power, hr, cad, temp)`.
- `validate_physics.py`:
  - `build_segments` — отрезки с факт. скоростью, уклоном, азимутом, мощностью;
  - `validate` — на каждый отрезок `v_pred = solve_speed(power, grade, headwind, mass, cda, crr)`,
    невязка `факт − прогноз`; агрегат: `mean_resid`, `rmse`, средние факт/прогноз;
  - `fit_cda_crr` — перебор `CdA∈[0.2,0.6]`, `Crr∈[0.003,0.008]` под минимум RMSE.
- Источники ветра: `era5` (archive-api.open-meteo.com, бесплатный реанализ), `const:W,DIR`,
  `snapshot:PATH` (наш `weather_now.jsonl`).

## Ограничения / на будущее

- Strava-GPX без мощности → нужен FIT/TCX/streams.
- ERA5 — почасовой и сглаженный; для тонкой сверки лучше наш `weather_now.jsonl` со дня заезда.
- Мощность отрезка — среднее концов; для точности можно интегрировать по всем точкам отрезка.
- Перебор `CdA/Crr` грубый (грид); при необходимости заменить на нормальную оптимизацию.
- Усталость (#4): смотреть дрейф невязки/требуемой мощности к концу длинных заездов.
