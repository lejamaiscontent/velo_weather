# Журнал решений и разборов

Почему сделано именно так + разборы накопленных данных. Новое добавляем сверху, с датой.

## Архитектурные решения (причины)

### Почему `fetch_weather` и `simulate_only` разделены
Каждый UI-action (смена км, мощность, стоянка) вызывает `simulate_only()` без запроса к API —
экономим лимит Open-Meteo (~10к запросов/сутки).

### Почему `_fetch_lock`
Три одновременных вызова `fetch_weather` на старте (startup + set_power + force_recalc) давали
120+ параллельных HTTP → 429. Блокировка вторых/третьих вызовов решила проблему.

### Почему нелинейная сетка точек
Ближние 100 км — каждые 4 км (детальный прогноз нужен скоро), дальше плавно до 25 км (экономим
запросы, прогноз всё равно теряет точность на 2–3 дня). Детали — в [weather.md](weather.md).

### Почему дешёвая проба (√N точек)
Полный фетч всех N≈52–114 точек на каждый поллинг — расточительно, когда чаще всего модель не
обновилась. Проба A=⌈√N⌉ (≤20) точек ловит факт обновления; полный фетч — только при изменении.
~6–10× экономия. Детали — в [weather.md](weather.md).

### Почему `_model_sched` не сериализуется
Состояние расписания осмысленно только в рамках сессии; после рестарта стартуем заново (первый
фетч помечается `new_run=true` — ложно-положительный, но безвредно).

## Анализ weather_log (17.06.2026, выборка крошечная — выводы предварительные)

Разбор 14 записей `weather_log.jsonl`, 16.06 21:38 → 17.06 08:32 UTC.

**Реальные обнаружения:**

| Модель | Detected UTC | Окно | Точность |
|--------|-------------|------|----------|
| ICON | 00:23 | 21:38→00:23 | ±2.75ч (выход из quiet) |
| ICON | 04:08 | 03:53→04:08 | **±15 мин (search)** |
| ICON | 06:53 | 04:08→06:53 | ±2.75ч (выход из quiet) |
| ECMWF | 07:38 | 21:38→07:38 | ±10ч (выход из quiet) |

**Выводы:**
1. Детекции садятся ровно на край quiet-зоны (00:23=21:38+2.75ч и т.д.) ⇒ реальное появление
   данных — *внутри* тихой зоны, латентность эти точки не измеряют.
2. Единственное настоящее измерение — ICON @ 04:08 (поймано в search): прогон 03Z D2/EU + ~1ч.
   Подтверждает: ICON обновляется по мелкой сетке каждые ~3ч, не только 00/06/12/18.
3. `quiet=2.75ч` для ICON, вероятно, длинновата (цикл D2/EU ~3ч) → едем на слегка устаревшем.
4. ECMWF при `quiet=10ч` не измерить — одна точка с окном 10ч.
5. `_weather_changed` даёт ложные негативы (грубая выборка пропускает обновления, слабо двигающие
   ветер в опорных точках).

**Методический итог (→ TODO #27):** текущая схема почти не измеряет истинную задержку публикации;
нужно сузить quiet либо принудительно входить в search вокруг расчётного `run+delay`, и логировать
каждый поллинг с ожидаемым временем прогона. Бо́льшую выборку теперь копят `weather_log` (с полями
`check`/`points`) и архив (#26).

## Известные ограничения
- Ветер на высоте 10 м, рельефного укрытия нет.
- Мощность постоянна (без учёта усталости — TODO #4).
- Осадки: линейная аппроксимация мм·ч.
- Всё в UTC.
- GPX без высот → рельеф не считается (предупреждение есть в `parse_gpx`).
- Первый фетч после рестарта всегда `new_run=true` (ложно-положительный).
