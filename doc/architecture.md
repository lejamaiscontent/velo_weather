# Архитектура и физика

## Физика (`core.solve_speed`)

```
P = [0.5·ρ·CdA·(v + v_wind)² + Crr·m·g·cos θ + m·g·sin θ] · v
```

Решается методом Ньютона относительно `v`. `v_wind > 0` → встречный ветер.
На спуске скорость ограничена (cap ~14 м/с / 50 км/ч), иначе Ньютон расходится.

## Направление ветра на отрезке (TODO #22 — корректно)

Встречная/попутная составляющая зависит от направления движения (азимута). Чтобы повороты
внутри 10-км отрезка не смазывались, азимут считается **покоординатно для каждого микросегмента
GPX** (пары соседних точек трека, ~каждые сотни метров), а не раз на 10 км:

- `parse_gpx` пишет `bearing_deg` для каждого микросегмента (`_bearing(a, b)`);
- `simulate` для каждого микросегмента берёт погоду в его точке/времени и считает
  `hw = effective_headwind(ws, wd, seg.bearing_deg)`, затем `solve_speed`;
- `effective_headwind` разворачивает метео-направление («откуда дует») на 180° и проецирует на
  азимут: `-wind·cos(wind_going − bearing)` (>0 встречный, <0 попутный);
- 10-км таблица (`grid_10km`) — лишь агрегация результатов микросегментов для отображения.

Огрубление, которое остаётся: погода берётся по ближайшей точке погодной сетки
(`_nearest_weather`) с интерполяцией по времени — это про разрешение прогноза, не про азимут.

## Планировщик остановок (`core.simulate`)

На каждом сегменте:
1. **Дождевая стоянка** — если `precip ≥ rain_threshold`, смотрим прогноз вперёд на
   `max_rain_wait_h`: если дождь кончится — ждём, потом едем.
2. Плановые стоянки — равномерно из бюджета `planned_stop_budget_h`; фактические
   (`actual_stops`) вычитаются из остатка.
3. Не превышаем `time_limit_h`.

(Ночёвка `overnight_*` удалена — TODO #7.)

### Дождь vs без дождя
Для каждой модели на каждом `simulate_only()` считаются **обе** симуляции; разница ТОЛЬКО в
`rain_threshold`: 0.5 мм/ч (с дождём) vs 9999 (без). Плановые стоянки одинаковы в обоих режимах.

### Стоп-баннеры в UI (шаблон)
- 🌧 `rain-wait` (оранжевый) — `stop_reason` содержит «дождь»;
- ☕ `planned` (серый) — любой другой `stop_reason` (плановый перерыв);
- 📍 `actual` (зелёный) — фактическая стоянка из `state.actual_stops`.

## Старт и время
- `route_start_time` (фикс. старт) передаётся в `simulate()` отдельно от `start_time` (текущая
  позиция); `elapsed_h` всегда считается от старта маршрута.
- Если `position_time ≤ route_start` — позиция игнорируется (старт ещё не наступил).

## Калибровка мощности (`core.calibrate_power`)
Бинарный поиск мощности по реальному `elapsed_h` езды (без стоянок) на отрезке
`from_km → to_km` от предыдущей зафиксированной позиции. Не перетирает мощность, заданную
вручную (`manual_power`). Входы и результат пишутся в `power_log.jsonl`.

## Поток данных в app.py
- `simulate_only(trigger)` — пересчёт по кешу погоды, без запросов к API; вызывается на любое
  пользовательское изменение и после фетча. В конце пишет снимок в архив (см. [archive.md](archive.md)).
- `fetch_weather` / `_fetch_models` — запросы к Open-Meteo (см. [weather.md](weather.md)).
- `_bg_loop` — умный фоновый цикл: проба → полный фетч при изменении; снимок текущей погоды раз в 30 мин.
