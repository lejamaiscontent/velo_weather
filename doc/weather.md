# Погода: источник, сетка, расписание, проба

## Источник
**Open-Meteo** (без ключа, бесплатно, прогноз ≤3 дня). Переменные:
`wind_speed_10m`, `wind_direction_10m`, `precipitation`, `temperature_2m`.
Бесплатный лимит ~10к запросов/сутки — отсюда вся экономия ниже.

## Сетка точек запроса (`core._sample_segments_nonlinear`)

Нелинейная, привязана к `current_km`:
- **ближние 100 км** (`NEAR_DIST`) — каждые **4 км** (`NEAR_STEP`);
- следующие **100 км** (`RAMP_DIST`) — шаг плавно растёт **4 → 25 км**;
- **дальше** — каждые **25 км** (`FAR_STEP`); финишная точка включена всегда;
- дубли по координатам (округление до 3 знаков, ~100 м) схлопываются.

Для текущих маршрутов **N ≈ 52–114** точек. Функция детерминирована по `(segments, current_km)` —
это важно для индексной сверки в пробе.

Зачем нелинейно: детальный прогноз нужен скоро (ближняя зона — частые точки), а на горизонте
2–3 дней прогноз всё равно теряет точность → дальше экономим запросы.

## Open-Meteo: расписание обновления моделей

| Модель | Прогоны (UTC) | Официальная задержка |
|--------|--------------|----------------------|
| ICON Global (DWD) | 00, 06, 12, 18 | ~2–3ч после прогона |
| ICON-EU (DWD) | каждые 3ч | ~1–2ч |
| ICON-D2 (DWD) | каждые 3ч | ~1–2ч |
| ICON seamless | комбо наиболее детальной доступной | ~1–3ч |
| ECMWF IFS 0.25° (open data) | 00, 06, 12, 18 | ~6–8ч суммарно |

**Для нас** (`icon_seamless` + `ecmwf_ifs025`): ICON реально обновляется ~через 3ч после прогона
(и по мелкой сетке D2/EU — каждые ~3ч), ECMWF — через ~6–8ч.

**Чего API НЕ отдаёт:** время инициализации модели (`model_init_time`); `generationtime_ms` —
это время HTTP-ответа, не прогона; `times[0]` в hourly — всегда 00:00 UTC текущего дня.
⇒ единственный способ понять, пришли ли новые данные, — **сравнить снепшоты**.

## Умное расписание фетча (`_should_fetch_model`)

**Если известно `last_update_at`:**
1. Тихая зона: не трогаем модель пока `age < quiet_h`.
2. После: поллинг каждые `poll_interval_min`.

**Если истории нет (рестарт/первый запуск):**
1. Медленная фаза: поллинг каждые `initial_interval_min`.
2. После `search_start_h` с первого фетча — быстрый поллинг каждые 15 мин.

| Параметр | ICON seamless | ECMWF IFS 0.25° |
|----------|--------------|-----------------|
| `quiet_h` | 2ч45м | 10ч |
| `search_start_h` | 2ч31м (151мин) | 11ч |
| `initial_interval_min` | 50 мин | 170 мин |
| `poll_interval_min` | 15 мин | 15 мин |

## Дешёвая проба обновления (`_probe_model`)

Чтобы проверка «обновилась ли модель» не стоила полного фетча всех N точек, цикл сначала делает
**пробу**: запрашивает только `A = min(20, ⌈√N⌉)` случайных точек (≈8–11) и сверяет их с теми же
индексами последнего полного снепшота. Экономия ~6–10× запросов на поллинг.

- Сетка строится при `last_weather_km` (current_km на момент снепшота) → индексы пробы и снепшота
  совпадают (`weather_query_segments` детерминирована).
- Результат: `False` — не обновилась, полный фетч пропускаем; `True` — отличие → полный фетч;
  `None` — нет снепшота / сетка разъехалась / ошибка сети → полный фетч.
- Полный фетч (`_fetch_models`) — только при `True`/`None`; он обновляет кеш и `last_update_at`.
  После «чистой» пробы кеш не трогаем.
- В quiet-зоне поллинга нет вообще; проба работает в search/slow-фазах.

### Детектор изменений (`_weather_changed` / `_diffs_exceed`)
Сравниваем точки, каждые 6ч прогноза: `max(diffs) > 0.5 м/с` по ветру **или**
`mean(diffs) > 0.15` (осадки с весом ×2). Порог с запасом: флуктуации аппроксимации ~0.1–0.2,
реальный новый прогон даёт >>1 м/с.

## Логи погоды

### `weather_log.jsonl` — каждая проверка
```json
{"t":"…","model":"icon_seamless","new_run":true,"mode":"quiet_2.75h",
 "check":"probe","points":8,"last_update":null}
```
- `check`: `probe` (дешёвая) / `full` (полный фетч); `points`: сколько точек запрошено.
  (Поля добавлены 17.06.2026; в старых записях их нет — читать `.get()`.)
- `mode`: `slow_50min`/`slow_170min` (нет истории), `search_15min` (ожидание прогона),
  `quiet_2.75h`/`quiet_10.0h` (тихая зона).

### `weather_now.jsonl` — стоп-гап текущей погоды (#30)
Раз в 30 мин (`_bg_loop` → `_snapshot_current_weather`) снимок ТЕКУЩЕЙ погоды по точкам маршрута
из актуального кеша (`core._interp(wp, now)`) — **нулевой расход API**. Копит непрерывный ряд
погоды-«истины» для местности. Подробнее — в [archive.md](archive.md).

## Почему `_model_sched` не сериализуется
Состояние расписания осмысленно только в рамках сессии. После рестарта первый фетч всегда
помечается `new_run=true` (нет снепшота для сравнения) — это нормально, мы и так получаем
свежие данные. Разбор накопленных логов — в [decisions.md](decisions.md).
